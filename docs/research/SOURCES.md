# TickTick research sources

Checked: 2026-07-18

This register contains direct primary sources used by docs/research/TICKTICK_FEATURES.md. Unless a row says otherwise, the source is an official TickTick product page, help-center article, policy page, or official app-store listing.

Confidence meanings:

- High: current official documentation directly supports the recorded behavior.
- High, bounded: current official documentation supports the behavior only for a named platform, plan, or workflow.
- Medium: official marketing/store material supports the claim but omits operational detail, or multiple official pages leave a boundary unclear.
- Low, historical: official but stale material; retained only to explain a conflict or former entitlement.

All pages were checked on 2026-07-18. A checked date confirms availability and content review, not a guarantee that TickTick will preserve the behavior.

## Known source conflicts and absences

### Attachment limits

Current official sources do not agree:

- S002 and current store descriptions advertise 99 attachments per day in Premium material.
- S102 describes 199 uploads per day, 39 attachments per task, and a 20 MB Premium file size.
- Free material consistently describes one upload per day and a 10 MB file size.

The ledger therefore treats the exact Premium daily and per-task cap as unresolved. It must not be copied into Omplish policy.

### Free-plan capacity

S006 historically listed nine lists, 99 tasks per list, two reminders per task, and 19 check items. Current official product/store pages continue to corroborate some Premium ceilings, but not all current Free ceilings. The historical figures are not current truth.

### Feature entitlement

Current public pages do not unambiguously classify Matrix, Kanban, Timeline, or the basic Pomodoro surface as Free or Premium on every client. Absence from the Premium page is not proof that a feature is universally Free.

### Offline and accessibility

No official source located during this review specified a complete offline mutation/conflict protocol, local encryption behavior, formal WCAG conformance, a VPAT, or complete screen-reader behavior. These remain evidence gaps rather than negative claims about the product.

### Platform parity

S003 establishes platform availability but not parity. Linux, Wear OS, VisionOS, and browser-extension details must be described only to the level supported by their dedicated documentation or store listing. Current Firefox extension availability was not verified.

## Research-use boundary

The official TickTick MCP endpoint documented by S609 may be useful for migration experiments or behavior checks against a sacrificial reference account. It is not a required Omplish dependency. Never connect personal production data merely to resolve a competitor research gap.
